# 워크플로우 — Project TODO 관리

> **대상 파일**: [`ai-docs/project_todo.md`](../project_todo.md)
> **목적**: 다영역(프론트/BFF/Engine/DB/인프라) 협업이 필요한 후속 구현 항목을 등록·진행·아카이브하는 일관된 절차 정의.
> **핵심 원칙**: `project_todo.md` 에는 **미착수(⬜) / 진행중(🚧) / 보류(⏸) 항목만 남는다.** 완료 항목은 즉시 제거하여 AI 컨텍스트 로딩 비용을 낮춘다.

---

## 트리거 시점

| 상황 | 해당 절차 |
|---|---|
| 후속 구현이 필요한 항목을 새로 발견했을 때 | [§1 신규 등록](#1-신규-항목-등록) |

## 1. 신규 항목 등록

1. **단일 영역 잔여 구현인지 먼저 판단**한다.
   - 화면 하나, 엔드포인트 하나로 끝나는 잔여 구현은 [`spec/frontend_features.md`](../spec/frontend_features.md) / [`spec/backend_features.md`](../spec/backend_features.md) 에 등록한다.
   - **프론트+백엔드+엔진 중 둘 이상** 손대야 하거나, 임시/데모 코드를 정식 흐름으로 승격해야 하는 항목만 `project_todo.md` 로 보낸다.
2. `project_todo.md` 에서 적합한 카테고리 섹션을 찾는다. 없으면 새 `## 카테고리명` 섹션을 추가한다.
3. 항목 형식:
   ```markdown
   ### ⬜ {항목 제목}

   **현 상태 ({YYYY-MM-DD})**
   - 무엇이 어떤 임시 형태로 구현/노출되어 있는지

   **정식 구현 시 필요한 작업**

   | 영역 | 항목 |
   |---|---|
   | Frontend | … |
   | BFF | … |
   | Engine | … |
   | DB | … |
   | QA | … |

   **참조 코드**
   - `path/to/file.ts:L1-L20`
   ```
4. 카테고리·소제목·상태 이모지(⬜)를 빠짐없이 적는다.

---

## 2. 착수 → 🚧

본격 작업으로 옮길 때:

1. [`task/active/${YYMMDD}_${title}.md`](../task/active/) 활성 태스크 파일 생성.
2. `project_todo.md` 의 해당 항목 상태를 `⬜` → `🚧` 로 변경하고, 활성 태스크 링크를 본문 상단에 한 줄 추가:
   ```markdown
   ### 🚧 {항목 제목}

   > 진행 중: [`task/active/260520_quest_auto_complete.md`](task/active/260520_quest_auto_complete.md)
   ```
3. [`context/current.md`](../context/current.md) 의 "활성 태스크" 라인에 동일 링크를 추가한다(GUIDELINE §2 SoT 규칙).
4. **항목 본문(현 상태 / 작업표 / 참조 코드) 은 그대로 둔다.** 활성 태스크 파일에서 다시 적지 않고 본 파일을 참조하면 된다.

---

## 3. 완료 → 아카이브 (핵심)

`project_todo.md` 가 비대해지지 않도록 **완료된 항목은 절대 남기지 않는다.**

### 3-A. 활성 태스크가 있던 경우

1. `task/active/${file}.md` → `task/${YYMMDD}/${file}.md` 로 이동(완료 일자 기준).
2. [`task/archive.md`](../task/archive.md) 의 해당 날짜 섹션에 한 줄 추가:
   ```markdown
   - [{항목 제목}]({YYMMDD}/{file}.md) — 한 줄 요약 (완료)
   ```
3. `project_todo.md` 에서 **해당 항목 블록 전체를 삭제**한다.
4. 카테고리 섹션이 비었으면 섹션 헤더도 제거.
5. [`context/current.md`](../context/current.md) "활성 태스크" 라인에서 해당 줄을 제거.
6. 영향 받은 명세는 갱신:
   - 프론트엔드 잔여 → `spec/frontend_features.md` 의 해당 항목을 ✅ 로 갱신
   - BFF/Engine → `spec/backend_features.md` 갱신
   - 사용자 시점 기능 → `/README.md` / `spec/overview.md` 도 GUIDELINE §4 에 따라 갱신
7. 외부 영향(엔드포인트, CSS 아키텍처, 위키 노출 항목 등) 이 있으면 [`wiki-update.md`](wiki-update.md) 절차를 함께 수행.

### 3-B. 활성 태스크 없이 인라인으로 끝낸 경우

작은 항목을 활성 태스크 파일 없이 즉시 해결한 경우:

1. `task/archive.md` 의 완료일자 섹션에 **1줄 등록 (인라인 표기)**:
   ```markdown
   - {항목 제목} — 한 줄 요약, 커밋 `abc1234` (인라인 완료)
   ```
   별도 상세 파일을 만들지 않는다.
2. `project_todo.md` 에서 항목 블록 삭제.
3. 카테고리 섹션이 비었으면 섹션 헤더 제거.
4. 명세 갱신은 3-A 6항 동일.

---

## 4. 보류 → ⏸

진행을 멈춰야 할 때:

1. 상태 이모지를 `⬜`/`🚧` → `⏸` 로 변경.
2. 본문에 보류 사유와 재개 조건을 명시:
   ```markdown
   ### ⏸ {항목 제목}

   > **보류 사유 ({YYYY-MM-DD})**: 정식 출시 전 보안 강화 사이클에 일괄 처리 예정.
   > **재개 조건**: 출시 D-30 진입 시 다시 ⬜ 로 환원.
   ```
3. 활성 태스크가 있었다면 `task/active/` 파일은 그대로 두고 본문에 보류 메모를 추가하거나, 닫고 추후 새로 생성하는 방향으로 정한다.

## 5. 점검 체크리스트

다음 시점마다 가볍게 훑는다:
- 큰 작업을 마치고 `current.md` 를 갱신할 때
- 주간 단위 청소 시점
- AI 컨텍스트 로딩이 무겁다고 느껴질 때

| 항목 | 점검 |
|---|---|
| `project_todo.md` 에 완료(✅) / 완료 의미 문구가 남아있지 않은가 | 남아있다면 [§3](#3-완료--아카이브-핵심) 절차로 즉시 제거 |
| 모든 `🚧` 항목 본문 상단에 활성 태스크 링크가 걸려있는가 | 없다면 활성 태스크 파일을 생성하거나 상태를 ⬜ 로 환원 |
| 비어있는 카테고리 섹션이 있는가 | 있다면 섹션 헤더 제거 |
| `⏸` 항목의 재개 조건이 여전히 유효한가 | 만료됐다면 ⬜ 로 환원 또는 항목 자체 삭제 |
| `current.md` 의 "활성 태스크" 라인과 `project_todo.md` 의 🚧 항목이 일치하는가 | 불일치 시 양쪽 동기화 |

## 6. 빠른 의사결정 표

| 상황 | 어디에 적나 |
|---|---|
| 단일 화면의 작은 잔여 구현 | `spec/frontend_features.md` |
| 단일 BFF 엔드포인트의 잔여 | `spec/backend_features.md` |
| 프론트+BFF (또는 +Engine) 동시 손봐야 함 | **`project_todo.md`** |
| 임시/데모 코드(예: DBG 버튼) → 정식 흐름 승격 필요 | **`project_todo.md`** |
| 지금 바로 착수 중인 큰 작업 | `task/active/${YYMMDD}_${title}.md` (+ project_todo.md 🚧) |
| 끝난 작업 이력 보관 | `task/archive.md` + `task/${YYMMDD}/` |
| 트러블슈팅 기록 | `trouble/${YYMMDD}/...` (별도 흐름) |
